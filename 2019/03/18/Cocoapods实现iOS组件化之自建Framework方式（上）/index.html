Cocoapods实现iOS组件化之自建Framework方式（上）
2019-03-18|iOS

前言

随着项目功能越来越多，工程也会越来越臃肿，为了更好的进行项目管理、测试和开发，工程就有了组件化的需求。如何将当前臃肿的工程逐步实现组件化就是这一系列文章将要解决的问题。简单来说，组件化即是将iOS功能模块打包成为Framework库文件，然后整合成为应用的过程。这一系列文章将对如何实现项目的组件化开发进行详细说明，包含诸多细节问题。

此上、下两篇文章是对自行建立Framework工程作为组建工程的方式进行说明，还有一篇文章是使用Pod打包Framework方式，有进行自动化脚本升级且便于调试，本人使用后面的方式实现组件化，如有需要可以跳转查看，但是通过这上、下两篇文章可以对内部的原理有一定理解。

本文Demo是Swift项目，使用pod对多项目进行管理。组件使用Framework形式，有Swift和OC混编。这一章节从创建Framework、pod管理组件、组件之间的沟通等方面进行统一的整理。
先看一下工程最终的效果。

工程框架搭建

组件化工程框架设计

组件化框架设计如上图，从下往上依次是库、服务（基础）、服务（应用）、业务（组件）、应用（App）

  • 库：应当将 库（本地三方）和 库（Cocoa）放于一个 repo 下，此项目不包含任何其他业务代码（纯粹扩展系统功能），相当于一个基础 framework 库，按照版本更新。具体就是包含 pod 和一些对应的扩展，部分分类亦可用于实现第三方接口发生变化时候更新三方库单维持原接口。（此部分API应稳定，故放置统一 repo）。
  • 服务（基础）： 将基础服务分别创建 repo，单独管理升级，将项目冠以 base 前缀。不应当依赖主工程，应当只依赖下层，即： 库
  • 服务（应用）： 分别创建 repo，单独管理升级，项目冠以 project 前缀。不应当依赖主工程，应当只依赖下层，即： 库
  • 业务： 根据情况依赖某些服务，分别创建 repo，项目冠以 module 前缀。不应当依赖主工程，应当只依赖下层，即： 服务
  • 应用： 主工程按需导入相应的应用，设置主框架，灵活插入组件即可实现应用更新迭代。

2、在主路径中新建一个Framework工程作为组件： Project - Framework & Library - Cocoa Touch Framework。创建完的Framework成会有一个 xxx.h 和一个 .plist 文件，xxx.h主要是入口，引入OC文件或pod的第三方使用(Framework也可以使用pod引入第三方，和平时项目方法相同，只不过在Profile文件使用use_frameworks!)

3、指定最低系统版本

4、Swift虽然已经支持静态Framework，但业务组件中有资源文件，静态Framework无法获取其中的资源，所以组件库使用默认的Dynamic Library Framework。

组件之间的问题处理

工程组件间的类和方法引用

1、对于Framework组件中的OC代码，为了使其他工程组件能够使用，可以按照上文中提到的 swift类调用OC类 中的设置方法对OC文件进行设置。

2、对于Framework组件中的swift代码，允许其他工程组件调用的类和方法前需要用关键字 public 或 open 声明。public 可以被任何人访问，但其他module中不可以被override和继承，而在module内可以被override和继承；open 可以被任何人使用，包括override和继承。

3、使用时，在OC文件中 #import 导入对应Framework的头文件；在swift中 import 对应Framework的Module。

如Demo中：HTSSdk 中的OC类调用 EvalutionModule 中的Swit类的方法。

1、在 HTSSdkHelper.Swift 中将类和暴露的方法声明为 public，并且在OC需要使用的方法前添加 objc 关键字。

2、在OC文件中导入 EvalutionModule 的头文件，即可使用 EvalutionModule 中的方法。

壳工程调用组件工程

壳工程调用组件工程，和上述的组件工程调用组件工程类似，但是，真机编译需要在Embedded Binaries中引用壳工程中用到的组件工程的Framework和Bundle。这是因为，从ios8开始，苹果官方支持我们构建的dynamic framework，我们构建的这几个组件Framework都是dynamic frameworks，这种类型的framework它需要签名code-signed以及嵌入到我们的app，否则我们真机运行时就会奔溃。

Cocoapods的用途

  • Cocoapods整合所有的本地组件Framework和壳工程，已经帮助我们做好了相关的引用配置，使这些Framework可以在工作空间中相互引用。
  • 工作空间中的所有Framework都可以在Podfile中单独配置依赖的第三方库使用。
  • 将本地开发的Framework制作成Cocoapods的私有库，用pod管理所有工程组件。这部分内容在下一章节进行详细说明。
